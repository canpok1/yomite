# 設計・実装パターンルール

linterでチェックできない設計・実装パターンに関するルール。

## 1. インターフェース設計

- 公開APIが必要最小限か（不要な公開がないか）
- インターフェースが利用者にとって自然で使いやすいか
- 抽象度が適切か（過度な抽象化・抽象化不足）
- 依存関係の方向が適切か

## 2. エラーハンドリング

- エラーが適切に伝播されているか（握りつぶしていないか）
- 前提条件が満たされない場合に早期エラー終了しているか
- フォールバック処理を多用していないか
- エラーメッセージが原因特定に十分な情報を含んでいるか

## 3. テスト戦略

- テストが実装の詳細ではなく振る舞いを検証しているか
- テストケースが境界値・異常系を含んでいるか
- テストの独立性が保たれているか（他のテストに依存していないか）
- テストヘルパーやフィクスチャが適切に使われているか
