# 設計・実装パターンルール

linterでチェックできない設計・実装パターンに関するルール。

## 1. インターフェース設計

- 公開APIは必要最小限にすること（不要な公開をしない）
- インターフェースは利用者にとって自然で使いやすい設計にすること
- 適切な抽象度にすること（過度な抽象化・抽象化不足を避ける）
- 依存関係の方向を適切に保つこと

## 2. エラーハンドリング

- エラーは適切に伝播させること（握りつぶさない）
- 前提条件が満たされない場合は早期エラー終了すること
- フォールバック処理を多用しないこと
- エラーメッセージには原因特定に十分な情報を含めること

## 3. テスト戦略

- テストは実装の詳細ではなく振る舞いを検証すること
- テストケースには境界値・異常系を含めること
- テストの独立性を保つこと（他のテストに依存しない）
- テストヘルパーやフィクスチャを適切に活用すること
